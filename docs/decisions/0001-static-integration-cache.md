# ADR 0001: Static integration cache (clink pattern → all shells)

- **Status**: Accepted
- **Phase**: G (0.1.16 candidate, develop branch)
- **Supersedes**: the `eval "$(runex export <shell>)"` rcfile-line
  installation pattern shipped in 0.1.13–0.1.14
- **Authors**: ShortArrow, with ChatGPT (Claude Code) collaboration
  on alternatives analysis
- **Date**: 2026-05-10

---

## Context

After 0.1.14 shipped, a hand-check on WSL Arch Linux surfaced a
~1 s pause every time the user pressed Space:

```
~> ls<Space>           # prompt blanks for ~1 second, then redraws
```

`runex timings` showed `command_exists` cache misses costing ~110 ms
each through `which::which`'s 9p stat walk over the WSL-inherited
`/mnt/c/...` Windows PATH entries. That alone explained ~330 ms of
the latency. The remaining ~700 ms turned out to be more fundamental:
`bind -x` invokes the hook function on every Space press, the hook
function ran `'runex' hook ...` (PATH-resolved), and the user's
PATH had `~/.local/share/mise/shims` ahead of `~/.cargo/bin/runex`.

A direct binary call:

```
$ time runex hook --shell bash --line "ls" --cursor 2
real  0m0.002s
```

A shim-routed call:

```
$ time runex hook --shell bash --line "ls" --cursor 2     # via mise shim
real  0m0.474s
```

That's mise startup overhead per keystroke. The user worked around
it by writing `~/Documents/GitHub/dotfiles/bash/src/bash_myplug.sh`
to (a) strip Windows-side mise paths from `$PATH`, and (b) resolve
runex via `which -a runex | grep -v /mise/shims/ | head -1` then
pass that absolute path through `runex export bash --bin <abs>`.

The user's feedback was direct:

> UX がかなり悪い。bashrc に書く量が多い。Rust じゃない部分が多すぎる
> 長寿 OSS なので堅牢で将来的なメンテナンス性が高い設計が必要

In other words: the workaround works, but it's the wrong layer.
The shell integration's job is to invoke runex efficiently, and
runex itself should make that efficient — not require every user
to hand-write 25 lines of bashrc to undo a default that was wrong
for them.

## Decision

`runex init <shell>` for **bash, zsh, pwsh, and nu** writes a
**static cache file** to a stable on-disk location. The user's
rcfile/profile gets a one-line `source` that points at that
cache. Inside the cache file:

- The `runex` binary path is the absolute `current_exe()` value at
  the time `runex init` ran, **not** the bare string `runex`. This
  eliminates per-keystroke PATH resolution.
- A version + bin header (3 lines starting with
  `# runex-integration-version: <N>` /
  `# runex-bin: <abs path>` /
  `# Generated by ...`) lets `runex doctor` detect drift.
- An interactive guard (`case $- in *i*) ;; *) return 0 ;; esac`
  for bash/zsh, `if (-not interactive) { return }` for pwsh,
  `if $nu.is-interactive { ... }` for nu) early-returns when the
  cache is sourced by a non-interactive shell, so `bash -c
  '...'`-style invocations and CI scripts that happen to inherit
  the user's bashrc become no-ops.

**Clink keeps its existing `%LOCALAPPDATA%\clink\runex.lua`
install path and freshness probe.** The cache layout mirrors
clink's pattern (which has been in production since 0.1.13);
extending it to the other four shells gives runex a single,
shell-agnostic install/refresh/check contract.

### Cache layout (frozen)

| Shell | Cache path |
|---|---|
| bash  | `<XDG_CACHE_HOME>/runex/integration.bash` |
| zsh   | `<XDG_CACHE_HOME>/runex/integration.zsh`  |
| pwsh  | `<XDG_CACHE_HOME>/runex/integration.ps1`  |
| nu    | `<XDG_CACHE_HOME>/runex/integration.nu`   |
| clink | `%LOCALAPPDATA%\clink\runex.lua` (unchanged) |

`<XDG_CACHE_HOME>` resolves via `infra::env::xdg_cache_home_with`:
`$XDG_CACHE_HOME` → `$LOCALAPPDATA` (Windows) → `~/.cache`
(non-Windows) → `~/AppData/Local` (Windows fallback).

### Refresh triggers

A cache is regenerated when:

1. **`runex init <shell>`** is run explicitly. Always rewrites,
   so re-running picks up a new template, new abbreviation rules,
   or a relocated runex binary.
2. **`runex add <key> <expand>`** or **`runex remove <key>`**
   succeeds. Only existing caches are refreshed (no opt-in side
   effect for shells the user hasn't installed). Failures on
   individual shells become stderr warnings, never fatal — the
   add/remove itself stays successful.
3. Doctor never auto-refreshes. It surfaces drift via
   `integration:<shell>:cache` Outdated rows and tells the user
   to re-run `runex init <shell>`.

## Considered alternatives

### A. doctor WARN + docs/recipes only (ad-hoc)

The simplest possible response: write a `runex doctor` check
that detects mise shim shadowing and document a copy-pasteable
WSL-cleanup snippet in `docs/recipes.md`.

**Rejected.** Surfaces the problem without solving it. Every
WSL+mise user still has to maintain their own bashrc patch.
Violates the user's stated long-OSS-maintainability priority and
the Phase D principle (see ADR-001/predecessor in
`runex/tests/architecture.rs`) that runex absorbs work users
shouldn't have to do themselves. Also leaves the per-keystroke
overhead 100% intact when users lose track of their dotfile.

### B. `runex init` writes the absolute path directly into the rcfile

Keeps the `eval "$(runex export bash)"` shape but rewrites the
`runex` substring inside the rcfile to the absolute current_exe
path at init time:

```bash
# ~/.bashrc
eval "$('/home/who/.cargo/bin/runex' export bash)"
```

**Rejected** for three reasons:

1. **Dotfiles sync hostile.** Users who sync `~/.bashrc` between
   machines (Mac laptop, Linux server, WSL) end up with a path
   that's wrong on every machine except the one that ran `runex
   init`. We'd be smuggling host-specific data into a file users
   often track in a public dotfiles repo.
2. **Self-healing breaks.** If runex is reinstalled to a new
   location (e.g. cargo→homebrew migration), the rcfile silently
   keeps trying the old path. The shell falls into the literal-
   space fallback path; the user sees "runex stopped working"
   with no obvious cause. The current `eval $(...)` form
   self-heals because the next shell start re-resolves the bare
   `runex` token.
3. **Shell startup PATH lookup remains.** The `eval $(...)`
   subshell still does one PATH walk (= one mise-shim hit) per
   shell start, which on WSL is ~470 ms of latency before the
   first prompt. That's small per session but adds up across
   tmux/screen panes opened in bulk.

### C. `runex export <shell>` defaults `--bin` to current_exe (no init changes)

A minimal tweak: change the default of `--bin` from `"runex"` to
`current_exe()`. The rcfile line `eval "$(runex export bash)"`
stays the same; the *contents* it eval-sources change so the
hook function uses an absolute path.

**Considered, partially adopted as G3.** This solves the
per-keystroke hit (great), but two problems remain that the
static-cache pattern solves cleanly:

- **Shell startup still pays one PATH walk** to find `runex`
  before `eval $(...)` can run. Same ~470 ms on WSL+mise.
- **`bash -c` non-interactive contracts.** A CI script sourcing
  the user's rcfile pays the same `runex export bash` startup,
  registers the function definitions in its environment, and
  potentially `bind -x`'s the space key in its non-interactive
  child shells. The interactive guard added in G2 fixes this,
  but the guard has to be inside the sourced script — exactly
  what a static cache provides.

Adopted as part of G (G3 is the export-side default change), but
not sufficient on its own.

### D. Lazy bind via PROMPT_COMMAND (init at first prompt)

Only register the binding the first time the prompt is drawn:

```bash
__runex_lazy_init() {
  source <(runex export bash --bin "$(command -v runex)")
  bind -x '"\x20": __runex_expand'
  unset -f __runex_lazy_init
  # remove ourselves from PROMPT_COMMAND
}
PROMPT_COMMAND="__runex_lazy_init; $PROMPT_COMMAND"
```

**Rejected.** Bash-specific pattern. zsh's `precmd` hook is
similar but not identical; pwsh has no equivalent for
PSReadLine; nu's reedline binds are evaluated at config time,
not lazily. Going down this road means **a different shape of
init for each shell**, which is the opposite of what the user
asked for.

## Consequences

### Positive

- **Zero per-keystroke PATH lookup.** The hook function calls
  runex by absolute path. mise shim, augmented `/mnt/c/...`
  PATH, anything else that slows `which::which` is irrelevant.
- **Zero shell-startup PATH lookup.** rcfile sources a static
  file by absolute path. `runex` doesn't run at all during
  shell start.
- **Non-interactive shells become no-ops.** `bash -c '...'`
  scripts that inherit the user's rcfile see nothing.
- **Shell-agnostic install contract.** Adding a new shell to
  runex means: write a template, register the path extension in
  `infra::integration_cache::cache_path`, add a one-liner to
  `cmd::init`, and that shell gets `init` / `add` / `remove`
  refresh / `doctor` checks for free.
- **doctor surfaces drift.** Schema version mismatches, missing
  baked binaries, and legacy `eval $(...)`-style content all
  show up as `integration:<shell>:cache` Outdated WARNs with
  the actionable advice (`re-run runex init <shell>`) right in
  the detail string.

### Negative

- **`runex init` is now required to install integration.** In
  0.1.14 a power user could `eval "$(runex export bash)"` from
  their dotfiles without ever running `runex init`. That still
  works (`runex export <shell>` is unchanged in user-visible
  behaviour and is part of the public CLI surface), but they
  miss out on the cache (= same per-keystroke overhead they had
  before). doctor flags the case as Outdated and recommends a
  one-time `runex init <shell>` to opt in.
- **One more file in the user's filesystem.** Before, integration
  state lived in the rcfile and the `runex` binary; now there's
  also `~/.cache/runex/integration.<ext>` per shell. `runex
  doctor` shows where each one is.
- **Cache invalidation is a real concern.** If a user manually
  edits the cache file, the next `runex init` / `runex add` /
  `runex remove` overwrites their edit. The header explicitly
  says `do not edit` for this reason. A future `--no-refresh`
  flag could carve an exit path; not in scope for G.

### Neutral

- **`runex export <shell>` keeps working** and now produces the
  same content the cache file would (header + interactive guard
  + absolute bin baked). Power users running it in a custom
  flow get the same Phase G benefits without going through
  `runex init`.
- **`--bin runex` (explicit) still produces a bare-name script**
  for users who want PATH resolution preserved (multi-machine
  dotfile sync where each machine has runex at a different
  location). Phase G3 explicitly preserves this opt-out.

## Implementation contract

For long-term maintenance, these are the load-bearing invariants
future contributors must preserve:

1. **Cache path layout is frozen.** `<XDG_CACHE_HOME>/runex/
   integration.<shell-ext>`. Renaming or relocating requires a
   schema-version bump (see point 2) and a doctor migration
   message.
2. **Cache file format is versioned.** Header line one is
   `<comment-prefix> runex-integration-version: <N>`. Bumping
   `INTEGRATION_CACHE_VERSION` causes doctor to mark all
   pre-bump caches as Outdated and ask the user to re-run
   `runex init <shell>`. This is the only sanctioned way to
   change the cache file body shape.
3. **Adding a new shell is a known recipe.** See `docs/setup.md`
   for the contributor checklist; the test suite already enforces
   the snapshot + interactive-guard pin via
   `app::shell_export::tests` and `tests/shell_integration.rs`.
4. **Refresh on add/remove must stay best-effort.** A failed
   cache write must never propagate to the add/remove exit code.
   Stderr warnings only.
5. **Doctor must never auto-fix.** It identifies drift and tells
   the user how to fix it (almost always `runex init <shell>`).
   Auto-fixing would surprise users who had a reason to leave
   the cache stale (e.g. while debugging a template change).

## Verification

End-to-end pinning lives in `runex/tests/shell_integration.rs`
(bash, Linux only) and the per-module unit tests:

- `infra::integration_cache::tests` — cache_path resolution per
  shell + atomic write + symlink reject + parent-dir creation
- `infra::env::tests::xdg_cache_home_with_tests` — XDG fallback
- `infra::integration_check::tests::cache_freshness` — every
  doctor branch (Skipped × 2, Ok × 2, Outdated × 4)
- `cmd::add_remove::tests` — silent refresh on add, no-op
  preservation on zero-match remove
- `cmd::init::tests` — handle() end-to-end with `EnvHomeDir`
- `app::init::tests::integration_line` — quoting safety for
  every shell's source line
- `app::init::tests::integration_line_nu_security` /
  `integration_line_clink_security` — RLO/BOM/ZWSP/control
  char/`$` injection-safety on the cache path

Manual hand-check (WSL Linux, mise+cargo install both present):

```sh
runex init bash --yes
exec bash
ls<Space>          # → instant `lsd ` (no prompt blank)
runex timings      # → all cache hits, total ~ms
runex doctor       # → integration:bash:cache up-to-date
```
